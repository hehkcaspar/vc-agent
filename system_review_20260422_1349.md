# System Review — VC Agent Frontend (localhost:3001)

**Reviewer:** Claude Code (Playwright-driven UIUX/usability audit)
**Date:** 2026-04-22 13:49
**Build tested:** `main` @ `c8faee9` — frontend dev server on `localhost:3001`
**Viewports exercised:** 375 (mobile), 768 (tablet), 1280 (laptop), 1440 (desktop), 1920 (wide)
**Routes exercised:** `/portfolio`, `/portfolio/entities/{id}`, `/academic`, `/academic/scholars/{id}`, `/settings` (all 5 sub-sections)
**Evidence:** 30+ screenshots + raw JSON in `/tmp/vc-review/` (screenshots, `findings.json`, `final_findings.json`)

---

## Summary

The app is **functionally solid on desktop** with a distinctive editorial aesthetic (Playfair serif display, ALL-CAPS accent headings, color-coded status badges). Dark mode is well-implemented and toggles cleanly. The 404 state is polite and actionable. No page errors, no failed requests, no horizontal-scroll bugs at any tested width.

**The three biggest issues are:**
1. **Mobile (375 px) is effectively unusable** for every data-heavy view — portfolio cards truncate to 3–4 letters, Academic table drops 4 columns, entity-detail metadata collapses to unreadable word-per-line wraps.
2. **Entity cards are non-semantic `<div>` with onClick** — not keyboard-focusable, no middle-click / right-click / “Open in new tab”, invisible to screen readers as links. Every single entity-list interaction inherits this problem.
3. **Several WCAG quick-wins are open**: duplicate `<h1>`, unlabeled form radios, icon-only buttons with no `aria-label`, and the tablist does not implement roving-tabindex arrow-key navigation (keyboard users can’t cycle tabs with arrows).

Nothing here is shipping-blocking for a desktop-only internal tool, but mobile and a11y fixes would be cheap wins with outsized impact.

## 1. Aesthetic & Visual Design

### 1.1 Strengths
- **Consistent visual language.** Playfair Display serif for `h1`/`h2` paired with condensed ALL-CAPS accent labels reads as editorial / “hedge-fund memo” — fits the product category.
- **Status color-coding is legible and consistent.** Deal-stage pills (`PORTFOLIO`, `PROSPECT`, `DILIGENCE`, `PASSED`, `EXITED`) use saturated but non-fluorescent palette; MOIC badges follow the same chromatic language.
- **Dark mode is high-quality.** Backgrounds shift to warm charcoal (not pure black), text drops to off-white, and color accents (gold, coral, mint) retain identity. No visible z-fighting between translucent panels. Toggle is bidirectional (button label swaps `DARK MODE` ↔ `LIGHT MODE`).
- **Icon set is unified.** Everything chrome-side uses `lucide-react` SVGs — no emoji / entity glyphs leaking into UI. Hover action icons on entity cards (pencil / archive / trash) are crisp at 1×.

### 1.2 Issues
- **H1 duplication.** `/portfolio` renders two `<h1>` elements — `"VC Portfolio"` (page title) and an earlier `"VC Portfolio"` (sidebar brand). Search-engine / reader impact is minor for an internal app, but it’s a screen-reader heading-landmark bug and a trivial fix (demote sidebar to `<h2>` or a non-heading `<div>` with `role="banner" > .logo`).
- **Form placeholder ALL-CAPS is aggressive.** Inputs like `"NAME"`, `"STAGE"` as placeholder text in the Create Entity modal lose contrast against the label and read as shout-y. Consider sentence-case placeholders with the ALL-CAPS reserved for inline labels above the field.
- **Settings sub-section icons are oversized relative to the label text** on the left-rail navigation (`Funds`, `Legal Review Checklist`, …). The 20 px icon dominates the 14 px caption.
- **Workroom split pane lacks a drag-handle affordance.** The boundary between file tree / preview / chat is invisible; a thin hover accent (1 px primary on drag-hover) would communicate that it’s resizable, assuming resize is supported. (If it isn’t, consider adding it — the preview column feels cramped on 1280×800.)

## 2. Responsive Adaptation

Tested each route at 375 / 768 / 1280 / 1440 / 1920.

### 2.1 Desktop (1280+)
Layouts are comfortable. At **1920** there is unused whitespace on either side of the main column in `/portfolio` — could stretch the grid to 4-up cards, or add a max-width to prevent the rail collapsing visually. This is a preference, not a bug.

### 2.2 Tablet (768)
- Portfolio grid correctly collapses to 2-up.
- Sidebar remains open and eats ~200 px — the **mobile toggle should engage at 768, not only at 480**.
- Academic ranking table scrolls horizontally without any scroll hint. A `box-shadow` on the right edge of the first column (sticky name col) would telegraph overflow.

### 2.3 Mobile (375) — the single biggest usability concern

| Screen | Problem |
|---|---|
| `/portfolio` | Entity card titles truncate to ~3–4 chars. Tag chips (`PORTFOLIO`, MOIC) wrap to 4 lines. Primary stat “Round size” disappears under a card-action cluster that always-shows on touch. |
| `/academic` | Ranking table drops **Priority / H-Index / Updated / Actions** columns silently — user only sees Name + partial signal feed. |
| `/portfolio/entities/{id}` | Entity description wraps word-per-line (“Lorem / ipsum / …”). Founder chips overflow out of the header. Tabs row (`Workroom | Facts | Screening…`) overflows horizontally with no scroll indicator; 3rd tab onward is hidden. |
| Workroom chat | Filenames in the left tree are clipped to <10 chars. Chat input is legible but the preset dropdown button pushes the send button off-screen. |
| Create Entity modal | Textarea fits, but footer buttons (`Cancel` / `Create`) stack vertically without horizontal gap. |

- **`.mobile-toggle` hamburger works** (opens sidebar drawer — captured in `mobile_sidebar_open.png`) but it’s the only mobile affordance that’s actually responsive.
- There is **no horizontal viewport overflow** at any width (confirmed `document.scrollWidth === clientWidth` at 375/768/1280/1440). Layouts truncate content rather than introducing a scroll — this is safer than the alternative, but the truncation itself is the bug.

**Recommendation.** The product is clearly desktop-first (this is fine for an internal VC tool). Either declare that explicitly (a `min-width: 1024px` block-and-polite-message for narrower screens), or invest in responsive variants — half-measures (hamburger + desktop everything-else) are worse than either endpoint.

## 3. Functional Bugs / Issues

### 3.1 Bugs
- **Duplicate `<h1>` on every portfolio page** (see §1.2).
- **Tablist arrow-key navigation is missing.** Pressing `ArrowRight` while focused on the `Workroom` `role="tab"` leaves focus on the same element (confirmed in `final_findings.json`). W3C APG specifies arrow-key cycling within a tablist — LTR-keyboard users currently have to Tab through every tabpanel’s descendants to reach the next tab.
- **Academic > “Tasks” view duplicates Settings > Continuous Tasks.** Both render the same `TasksView` component, and actions in one silently affect the other. Consider consolidating to one location; the other can deep-link to it.
- **Entity edit modal opens behind a translucent overlay that intercepts pointer events** for some underlying buttons while the modal is open. This showed up in Playwright as the “Create” button on the portfolio page being matched under the modal’s “Create Entity” submit. Not a user-visible regression (the overlay is intended), but the `pointer-events: auto` on the overlay + `<button>` elements underneath means if the user presses `Tab` past the end of the modal they can focus buttons underneath it. Add `focus-trap` to the modal.
- **Create Entity validation is silent.** Submitting the modal with an empty `name` field produces no visible error, no toast, no field outline — the form just does nothing. Either the button should be `disabled` until name is non-empty, or the field should bounce with an error message.

### 3.2 Not-a-bug but worth confirming
- **Entity detail URL uses `?chat={uuid}`** (e.g. `/portfolio/entities/abc?chat=def`). Good pattern for session reattach, but the chat-session query param is not preserved across tab switches within the entity detail (Workroom → Facts → back to Workroom drops the `?chat=`). Verify this is intentional.

## 4. Usability

- **Entity cards are `<div onclick>`, not anchors.** Consequences the user will hit:
  - `Cmd/Ctrl+click` doesn’t open in a new tab.
  - Right-click has no “Open link in new tab”.
  - Keyboard tab focus skips them entirely (they have no `tabindex`).
  - `Copy link address` is unavailable.
  - Screen readers don’t announce them as navigable.

  **Fix.** Wrap each card body in `<a href="/portfolio/entities/{id}?chat={last_session}">` (or use a `<Link>`), then attach any mouse-down handlers to the anchor. Preserve the card-action hover buttons (edit / archive / delete) as real nested `<button>`s with `stopPropagation()`.

- **Icon-only actions lack tooltips and aria-labels equally.** The hover reveal on entity cards exposes Edit / Archive / Delete icons. They have `title="Edit entity"` etc. (good for a slow-hover mouse user) but `aria-label` is null — screen readers get nothing. Add `aria-label="{Edit|Archive|Delete} {entityName}"`.

- **Sidebar collapsed state shows icons only with no labels.** When the sidebar is collapsed, the nav entries lose their text; icons alone (`House`, `GraduationCap`, `Settings`) are recognizable but there are no `title` / `aria-label` attributes, and no tooltip on hover. Either keep labels visible in collapsed state (narrow + short) or add proper tooltips.

- **Portfolio “Health 49%” metric has no explanation.** Users will want to know what it measures. Add a `?` tooltip or link to docs.

- **Empty-state handling for “no matching entities”** after filtering is a plain text line (`No entities match these filters.`). Consider adding a subtle CTA (“Clear filters” button) — it’s a two-line change but meaningfully reduces dead-ends.

- **Global search is absent.** For a tool that will accumulate hundreds of entities + scholars, a `Cmd+K`-style quick nav is a high-leverage add. Not in scope for this review, noting as a backlog item.

## 5. Accessibility (WCAG AA)

### 5.1 Contrast
Sampled contrast in both themes (see `final_findings.json`). **No WCAG-AA failures** found in the sampled headings, buttons, tabs, inputs, or body text. Muted helper text (`.text-muted`) sits at ~5.1:1 in light mode and ~6.3:1 in dark — within spec.

### 5.2 Non-text issues
- **Form radios in the Entity Edit modal have neither `<label for=>` nor `aria-label`.** Stage-selector radios expose only visible glyph labels. Add `<label>` wrapping.
- **`<input id="name">` in Create Entity modal has no associated `<label>`.** Only a placeholder, which isn’t an accessible label. Add a `<label>`, even if visually hidden.
- **`.mobile-toggle` hamburger has no `aria-label`.** Should be `aria-label="Open navigation"` with `aria-expanded` toggling.
- **Duplicate `<h1>`** breaks landmark navigation (see §3.1).
- **Entity cards lack `role="button"` / `tabindex="0"`.** (Prefer the `<a>` conversion above over ARIA patching.)
- **No visible focus ring on primary `<button>` elements.** Computed `outline: "... none 0px"` on most buttons; only `role="tab"` elements get a 2 px solid outline. Add `:focus-visible` styling globally.

### 5.3 Positive
- Tabs use `role="tab"` correctly.
- Images have non-empty `alt` text where observed.
- Modals are dismissible by `Escape` (confirmed via keyboard scripting).
- No reliance on color alone for state — status pills pair color with text.

## 6. Console & Runtime

- **Zero `pageerror`, zero `requestfailed`, zero 5xx responses** across every route exercised.
- **React Router v7 future-flag warnings** surface on every navigation:
  - `v7_startTransition`
  - `v7_relativeSplatPath`

  These are advisory, not breaking. Opt in explicitly in the router config to silence:
  ```ts
  createBrowserRouter(routes, { future: { v7_startTransition: true, v7_relativeSplatPath: true } })
  ```

## Update — 2026-04-23

Most of the prioritized fix list has shipped. Changes since the original review:

**Shipped:**
- **#1 Entity cards as anchors** — `EntityCard` / `EntityRow` in `PortfolioTab.tsx` are now `<Link>` with `aria-label` + preventDefault-wrapped action buttons.
- **#2 Mobile posture** — chose Tablet+ (≥768). `ViewportGuard` gates the router; `<768` shows a polite denial with current-width readout, theme-correct via synchronous `data-theme` set in `main.tsx`. All mobile CSS breakpoints aligned to `max-width: 767px` so exactly 768 lands in the tablet (collapsed-sidebar) layout. Ranking table got `min-width: 720px` + visible thin scrollbar so the 4 dim columns stay reachable.
- **#3 Labels + aria-labels** — mobile toggle / theme toggle / sidebar collapse / entity card action buttons / scholar chat textarea / entity chat textarea all carry `aria-label`. Form inputs were already correctly labeled — false positive in the original audit.
- **#4 Roving tabindex** — `EntityDetail` tablist now supports `ArrowLeft`/`ArrowRight`/`Home`/`End` via a post-commit `useEffect` keyed on `pendingTabFocusRef`.
- **#5 Single h1 per page** — sidebar + mobile-header brand demoted to `<div>`; PortfolioTab / AcademicTab / ScholarDetail page headings promoted to `<h1>`; the ReportTab narrative headline in `ScholarDetail.tsx` demoted from `#` to `##` so hierarchy is scholar-name h1 → narrative-headline h2 → sections h3.
- **#7 Tasks consolidation** — Academic tab toolbar is now List | Ranking only; `TasksView` lives under Settings > Continuous Tasks; legacy `/academic?view=tasks` redirects via `navigate(..., { replace: true })`.
- **#8 Focus-visible ring** — global `:where(a, button, input, ...):focus-visible` outline in `primitives.css` using `--color-accent-gold`.
- **#9 Router v7 warnings** — `main.tsx` opts into `v7_startTransition` + `v7_relativeSplatPath`.
- **Bonus — Modal focus trap** — `components/ui/Modal.tsx` now traps Tab/Shift+Tab inside the dialog, lands initial focus on the first non-close focusable, and restores focus to the trigger on close (no external library). Addresses the §3.1 "modal overlay pointer-events" concern.
- **Bonus — Settings + ScholarDetail a11y sweep** — every Settings sub-page + every scholar detail tab verified clean: `h1=1`, no unlabeled inputs, no unlabeled icon buttons, no broken tablists.

**Retracted (false positives in original audit):**
- "Create Entity validation is silent" — the submit button was already disabled with `!metadata.name?.trim()`.
- "Form radios lack labels" — radios wrap their `<input>` in `<label>`, which is accessible; the audit only queried `label[for=id]`.

**Still open:**
- #6 mobile responsive rebuild — intentionally not done; Tablet+ posture makes `<768` out of scope.
- #10 "Health %" tooltip — couldn't locate the metric in portfolio code; the audit surfaced it only in the scholar/tasks `TasksView`.

*End of review.*
